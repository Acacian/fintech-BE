# ADR-001: MSA 서비스 분리 기준

## Status
Accepted

## Context
팀은 재직자 5인으로 구성되어 있었고, 각자 기술 배경과 관심 도메인이 달랐다.  
초기에는 한 저장소 안에서 기능을 빠르게 붙이는 방식이 가능했지만, 결제 기능이 커질수록 변경 충돌이 늘었고 특정 기능만 따로 배포하기도 어려워졌다.  
가맹점 온보딩, 결제 처리, 사용자 관리, 백오피스 조회 기능은 변경 주기와 책임 주체가 달랐기 때문에 서비스 경계를 먼저 분명하게 나눌 필요가 있었다.

## Related Issues
- [Issue #1](https://github.com/Acacian/fintech-BE/issues/1)

## Decision
저장소는 하나로 유지하되, 도메인 책임 기준으로 아래 6개 모듈로 분리한다.

- `backoffice-api`: 관리자/백오피스 조회 API
- `backoffice-manage`: 가맹점 가입, 로그인, SDK Key 관리
- `payment`: 결제 준비, 진행 조회, 실행, 취소
- `appuser-manage`: 앱 사용자 가입, 인증, 카드/거래 관리
- `payment-method`: 결제수단 도메인 공유 라이브러리
- `common`: JWT, 예외, 로깅 등 공통 모듈

## Consequences
(+) 가맹점, 사용자, 결제, 조회 기능의 책임 경계가 명확해진다.  
(+) 서비스별 독립 실행과 개별 Docker 이미지 빌드가 가능해진다.  
(+) 도메인 분리 기준과 팀 단위 협업 구조를 명확하게 설명할 수 있다.  
(-) 서비스 간 계약과 공통 모델 관리 비용이 늘어난다.  
(-) 공유 모듈 변경이 여러 서비스 빌드에 연쇄 영향을 줄 수 있다.  
(-) 배포 단위가 늘어나 운영 복잡도가 증가한다.
